## States Covered

- `Default`: Inputs and buttons are enabled. Invite code field is prefilled with `SEED-XIAOYU-001`.
- `Loading`: Primary button and secondary actions are disabled; primary button text changes to `正在进入...`.
- `Error`: A specific, recoverable error appears near the relevant input; fields are re-enabled.
- `Success`: A brief success message appears and the prototype navigates to `01.2-home-dual-entry.html`.

## What to Build

### HTML Structure

Add a primary action block after `#start-login-method-panel` and before `[data-region="boundary-note"]`.

Required:

- Add a button:
  - `id="start-login-primary-button"`
  - `data-object-id="start-login-primary-button"`
  - `type="button"`
  - visible text `进入 Style AI`
- Add a status text element:
  - `id="start-login-action-status"`
  - hidden by default
  - used for success/loading helper messages, not field-specific errors
- Keep existing field-level error elements:
  - `#invite-code-error`
  - `#phone-login-error`

Suggested structure:

```html
<section
  class="rounded-lg border border-white/10 bg-white/[0.045] p-4"
  data-region="login-actions"
  aria-label="Login actions"
>
  <button
    id="start-login-primary-button"
    data-object-id="start-login-primary-button"
    type="button"
  >
    进入 Style AI
  </button>
  <p id="start-login-action-status" hidden></p>
</section>
```

The primary action may also be placed visually as part of the existing login panel if that better preserves mobile rhythm, but the `data-region="login-actions"` wrapper and object ID must be present.

### Tailwind Classes to Use

Key classes:

- Action shell: `rounded-lg border border-white/10 bg-white/[0.045] p-4`
- Primary button: `min-h-[48px] w-full rounded-lg bg-[#FAFAF7] px-4 py-3 text-sm font-semibold text-[#111111]`
- Button transition: `transition active:scale-[0.99]`
- Disabled state: `disabled:cursor-not-allowed disabled:opacity-55`
- Status text: `mt-3 text-xs leading-5 text-white/55`
- Error text: existing `text-red-200`; keep errors near their field
- Success text: use `text-[#C9BAFF]` or `text-white/72`, not a large success banner

Do not make the action look like a marketing CTA. It should read as a mobile app login control.

### JavaScript Requirements

Add prototype-only state helpers. Keep them small and local to `01.1-start-login.html`.

Required constants:

```javascript
const demoPhoneNumber = '13800000001';
const demoPhoneCode = '123456';
const homeTargetUrl = '01.2-home-dual-entry.html';
```

Required functions:

- `setLoginActionState(state, message)`
  - `state` values: `default`, `loading`, `error`, `success`
  - updates button text and disabled state
  - disables/enables invite, phone, code, get-code, Google controls
  - writes status message to `#start-login-action-status`
- `showFieldError(target, message)`
  - `target`: `invite` or `phone`
  - writes to `#invite-code-error` or `#phone-login-error`
- `handlePrimaryLogin()`
  - clears prior messages
  - if phone number or verification code has input, validate phone/code first
  - if phone fields are blank, validate invite code
  - if both paths are blank, show invite error: `请输入邀请码，或使用手机号 / Google 登录。`
  - on valid invite code, call success path
  - on valid phone and code, call success path
  - on invalid input, show field-specific error
- `handleGetPhoneCode()`
  - if phone is missing, show phone error: `请输入手机号后获取验证码。`
  - if phone is present, show status text: `验证码已发送，原型验证码为 123456。`
  - no real SMS call
- `handleGoogleLogin()`
  - shows loading state
  - mock success when `googleLogin.enabled === true` in demo assumptions
  - then calls success path
- `completeLogin()`
  - shows success text: `登录成功，正在进入首页。`
  - navigates to `01.2-home-dual-entry.html` after a short delay

Validation rules for prototype:

| Path | Valid Input | Invalid Feedback |
|------|-------------|------------------|
| Invite code | `SEED-XIAOYU-001` | `邀请码无效或已过期，请检查后重试。` |
| Phone | phone `13800000001` and code `123456` | `手机号或验证码不匹配，请重新输入。` |
| Google | click Google button | Mock success; future backend can reject unauthorized accounts |

Implementation detail:

- Use `window.setTimeout` for short loading/success transitions.
- Keep delays short, around 400-700ms, so the prototype remains easy to test.
- If `01.2-home-dual-entry.html` does not exist yet, navigation can push the target URL with `history.pushState`. That is acceptable for this section as long as the URL target is correct.
- Update `pageMeta.section` to `4-login-actions-and-states`.

## Common Issues & Fixes

### Issue: Success Navigates Too Fast

**Symptom**: User cannot perceive success state.  
**Cause**: Immediate `window.location.href` change.  
**Fix**: Keep a short 500-700ms success delay before navigation.

### Issue: Phone Path Competes With Invite Code

**Symptom**: Primary button validates phone unexpectedly while invite code is still populated.  
**Cause**: Invite code default remains present.  
**Fix**: If phone number or verification code has input, treat phone login as the active path; otherwise validate the prefilled invite code.

### Issue: Missing Next Page Looks Like a Login Failure

**Symptom**: Success navigates to a 404 because `01.2-home-dual-entry.html` is not built yet.  
**Cause**: Later page not implemented.  
**Fix**: Explain in verification that URL target is the required behavior for this section. Do not build the next page here.
