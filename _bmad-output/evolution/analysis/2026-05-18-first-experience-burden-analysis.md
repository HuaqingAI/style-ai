# 首次体验用户减负分析

**项目:** style-ai  
**日期:** 2026-05-18  
**活动:** WDS-8 Product Evolution / Analyze Product  
**关注范围:** Scenario 01 `小雨的首次真实预览`、DD-001 本地 HTML 原型、当前 Expo 应用首屏与结果页；附带检查 Scenario 02/03 是否把首次减掉的负担重新带回  
**目标用户:** 小雨 - 主动想变美但缺少判断力的新手

---

## 机会定义

首次体验当前验证目标是：20-50 个种子用户中，至少 60% 完成首次个人形象生成或试穿生成，并表示结果“像自己、可用于判断”。

用户给出的判断方向是成立的：首次体验真正要卖的是“快速看到一个像自己的结果”。照片质量检查、生成设置、保存档案、反馈、历史、额度等能力都有产品价值，但在首次链路中过早显性化，会把系统内部流程和长期管理负担转嫁给用户。

---

## 产品快照

### 已有 WDS 设计

Scenario 01 当前定义为 7 个显性步骤：

1. 启动/登录页
2. 首页/双入口页
3. 照片上传/拍摄页
4. 照片质量检查页
5. 个人形象生成设置页
6. 生成等待页
7. 个人形象结果页

这些页面已经完成文本规格和本地 HTML 原型；DD-001 已通过本地原型验收。验收覆盖了 `01.1 -> 01.7` 的 happy path、移动端宽度、结果页交互、反馈提交和无横向溢出。

### 当前 Expo 实现

真实 Expo app 仍是早期“照片分析”形态：

- 首页直接提供拍照/相册、分析分类选择和“开始 AI 分析”。
- 结果页展示上传照片、综合评分、summary、aspects 和 suggestions。
- 代码模型是 `AnalysisResult`，还没有 DD-001 需要的生成图、推荐方向、相似度反馈、可判断反馈、额度、任务状态、历史/档案边界等结构。

这意味着后续开发不能简单把 7 个 HTML 原型页原样搬到 Expo。更好的方向是在实现前先重新压缩首次体验结构。

---

## 当前体验负担

### 1. 首次链路过长

原型的 happy path 是 `进入 -> 首页 -> 选照片 -> 质量检查 -> 生成设置 -> 等待 -> 看结果`。其中“质量检查”和“生成设置”对系统很重要，但对首次用户来说更像额外审批和配置。

用户此时的真实任务不是“把参数配置正确”，而是“先看看这个工具能不能让我看到自己”。每增加一步，都会增加犹豫、退出和自我怀疑。

### 2. 系统语言外露过多

现有规格和原型中反复出现“质量检查、生成设置、额度、常用形象、反馈、历史”等概念。这些概念在产品机制上正确，但首次用户会把它们理解为要学习和管理的新系统。

更适合首次体验的语言是：这张能不能参考、哪里不像、换个方向看看、保存这个结果、下次少传一次。

### 3. 默认生成仍然要求用户决策

`01.5-generation-setup` 虽然默认有“AI 推荐”，但页面仍显性展示五个方向、可选说明、备注输入和额度说明。它把“系统可以先给一个稳妥方向”的能力表达成了“用户先来选一个方向”。

首次用户缺少判断力，正是产品要帮助她的地方。首轮应默认化，不应让她先证明自己会选。

### 4. 照片质量检查被做成了独立关卡

`01.4-photo-quality-check` 对降低失败率有价值，但作为单独页面，会让可用照片也被迫经历一次审核感。轻微问题适合在上传页或等待前用一句提示处理，只有严重不可用才阻断。

### 5. 结果页承载目标过多

`01.7-image-result` 的方向已经比早期报告式体验更好：主图、判断、可展开详情、轻反馈都存在。但验收记录显示结果页同时有推荐切换、详细分析、像不像/能不能判断、issue tag、profile prompt、保存、换风格等模块。

首次结果的第一屏应该只帮助用户完成一个判断：这张结果能不能参考。其他能力应渐进出现。

### 6. 反馈和建档仍然太像任务

反馈入口如果默认呈现完整面板，会变成问卷；建档如果在第一次结果页强推，会变成管理负担。它们应该由用户行为触发：

- 点“不太像 / 不好判断”后再展开具体原因。
- 点“保存”、第二次生成、或准备试穿时，再询问是否保存为常用形象。

## 根因

原始设计为了回答种子验证中的信任问题，把每个信任风险都做成了显性说明或显性步骤：

- 怕照片被乱存 -> 上传前解释历史和常用形象边界。
- 怕生成不像 -> 单独质量检查和结果反馈。
- 怕扣费不清楚 -> 首页、设置页、等待页都解释额度。
- 怕结果不可判断 -> 结果页加入推荐、谨慎点、详情、反馈。

这些都是正确风险，但被平均铺到主路径后，主路径从“快速看到自己”变成“完成一套产品教育”。首次体验应把风险处理改成分层策略：默认路径最短，只有在用户需要、系统异常、或用户主动表达意图时才展开更多说明。

---

## 改进目标优先级

| 优先级 | 目标 | 影响 | 工作量 | 学习价值 |
|---|---|---|---|---|
| 1 | 压缩首次生成主链路为 `进入 -> 选照片 -> 等待 -> 看结果` | 高：直接影响首次生成完成率 | 中：需要重写 01.3-01.7 的流程规格和原型路由 | 高：验证“短链路是否提升完成率和结果判断率” |
| 2 | 将生成设置默认化，改为结果页“换个方向” | 高：减少首次选择压力 | 中 | 高：验证默认推荐是否足以启动二次探索 |
| 3 | 将照片质量检查降级为上传页轻提示/严重阻断 | 高：减少审核感 | 中 | 中：验证失败率和完成率之间的平衡 |
| 4 | 重排结果页第一屏，只保留结果图、一句判断、两个动作 | 高：强化首次价值感 | 中 | 高：验证用户是否先看结果再探索 |
| 5 | 将反馈改为反应按钮，按需展开原因 | 中：保留学习信号但减少问卷感 | 低到中 | 中：验证反馈量是否下降、质量是否提升 |
| 6 | 保存档案后置到保存后/第二次生成/试穿前 | 中：降低首次管理负担 | 低到中 | 中：验证档案创建转化时机 |
| 7 | 全链路文案从系统语言改为用户语言 | 中：降低认知成本 | 低 | 中：验证可理解度和信任感 |

## 本轮选定目标

**IMP-001: 首次生成短链路减负**

把 DD-001 的首次路径从 7 个显性步骤压缩为 4 个用户感知步骤：

1. 进入
2. 选照片
3. 等待
4. 看结果

本轮不删除质量检查、生成设置、反馈、历史、档案、额度能力，而是重新定义它们的显性程度：

- 照片质量：自动检查；可用直接继续；轻微问题提示；严重不可用才阻断。
- 生成方向：默认生成“适合日常判断的稳妥方向”；结果页提供“换个方向”。
- 结果页：第一屏优先结果图、一句判断、换个方向/保存两个动作。
- 反馈：默认是轻反应；负向/不确定时展开原因标签。
- 常用形象：保存后、第二次生成、或进入试穿前再提示。
- 额度：首次免费/剩余额度不在主路径反复解释；失败/不足/购买场景再显性处理。

### Party Mode 评审采纳同步

2026-05-18 的 BMAD Party Mode 评审已确认并采纳本方向。采纳后的范围约束是：

- `01` 是主修改范围，目标是压缩 DD-001 首次可见流程。
- `02` 不进入大改，但必须做负担回流检查，避免“换个方向”重新变成复杂设置页。
- `03` 不进入大改，只做 profile 保存时机兼容；不能把常用形象档案变成试穿前置门槛。
- `01.4-photo-quality-check` 和 `01.5-generation-setup` 从首次 happy path 移除，但其能力不删除，分别下沉为照片质量状态、默认生成策略、异常分支和埋点事件。
- 设计原则从“删步骤”修正为“压缩可见流程，不压缩系统状态”。

评审提出的关键设计待决问题：

1. 第一次自动生成的默认方向是什么，如何解释？
2. 严重照片阻断标准是什么？
3. 结果页一句判断由 AI、规则、模板，还是组合方式生成？
4. `换个方向` 是直接生成，还是先给少量轻选项？
5. `保存` 保存的是图片、结果、风格方向、偏好，还是 profile？
6. 负向反馈后只是记录，还是触发重生成/修正方向？
7. `02` 如何保持低配置，避免成为隐藏版 `01.5`？
8. `03` 在没有 saved profile 时的最小可用体验是什么？

## 成功标准

本轮分析建议后续 Scope/Design 阶段以这些指标判断：

- 首次 happy path 的用户感知步骤不超过 4 步。
- 从选中照片到生成任务创建不需要用户选择风格或填写偏好。
- 只有严重照片问题会阻断；轻微问题不新增页面。
- 结果页首屏只出现一个主结果、一句可判断结论、两个主动作。
- 反馈默认入口不呈现完整问卷。
- 常用形象不在第一次结果首屏强推。
- 测试目标仍绑定原业务指标：首次生成完成率和“像自己 / 可用于判断”反馈。

### 原型验收约束

- `AC-DD001-01`: 新用户从登录到首张结果，主路径不得出现独立质量检查页。
- `AC-DD001-02`: 新用户从登录到首张结果，主路径不得出现生成参数设置页。
- `AC-DD001-03`: 上传合格照片后自动进入等待页。
- `AC-DD001-04`: 轻微照片问题不阻断生成，只显示 inline 提示。
- `AC-DD001-05`: 严重照片问题阻断，并给出单一明确动作：重新上传。
- `AC-DD001-06`: 结果页首屏必须包含结果图、一句判断、`换个方向`、`保存`。
- `AC-DD001-07`: 反馈默认不展开；负向或不确定反馈后才展开原因。
- `AC-DD001-08`: 保存、第二次生成、try-on 前才触发资料保存，不在首轮生成前强制出现。
