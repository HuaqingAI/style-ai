# 03.4.1 Tryon Waiting Full Page

## Purpose

Build `03.4-tryon-waiting.html` as a mobile prototype draft for the asynchronous waiting state after Linman starts a try-on generation.

## Objects

- `obj-waiting-header`: top navigation with back path to `03.3-tryon-setup.html`.
- `obj-waiting-hero`: concise explanation that her selected reference is being generated onto her profile.
- `obj-waiting-status`: progress panel with circular progress, status chip, current copy, task ID, and estimate.
- `obj-waiting-context-summary`: selected reference, profile image, intent, and decision question.
- `obj-waiting-credit-policy`: credit policy card that changes copy and tone by state.
- `obj-waiting-processing-steps`: four-step processing list with done / active / waiting / blocked states.
- `obj-waiting-recovery-actions`: state-specific actions.
- `obj-waiting-state-controls`: local reviewer controls for six page states.

## State Behavior

- `queued`: task created, 10% progress, success-only credit copy.
- `generating`: preview generation in progress, 48% progress, still no credit deducted.
- `finalizing`: result almost ready, 82% progress, organizing judgment notes.
- `success`: 100% progress, credit consumed after success, CTA to `03.5-tryon-result.html`.
- `timeout`: longer wait, not failed, no credit deducted yet, continue waiting or return to setup.
- `failed`: generation failed, no credit consumed, retry or return to setup.

## Layout Requirements

- Mobile-only single column, max width `428px`.
- Use the same Tailwind CDN setup and dark Style AI surface as `03.2` and `03.3`.
- Circular progress and progress bar must have stable dimensions at 375px.
- Cards use `rounded-lg`, light borders, compact mobile text scale.
- Avoid horizontal overflow at 375px.

## JavaScript Requirements

- Implement `setWaitingState(state)`.
- Render processing steps from state configuration.
- Render state-specific recovery actions.
- Route the `03.5` result action to `03.5-tryon-result.html`.
- Expose `window.__STYLE_AI_PROTOTYPE__` with state setter for automated verification.

## Acceptance Criteria

Agent-verifiable:
- All object IDs exist.
- Six states can be triggered.
- Progress text and progress bar update for each state.
- Processing step statuses update for each state.
- `success` action references `03.5-tryon-result.html`.
- `failed` state includes copy indicating failure did not consume credits.
- No console errors at load and during state switching.
- No horizontal overflow at 375px, 393px, and 428px.

User-evaluable:
- It is clear the app is working on the exact reference and profile chosen in previous steps.
- The credit rule feels explicit without feeling punitive.
- Timeout and failure states reduce anxiety and give obvious next actions.
