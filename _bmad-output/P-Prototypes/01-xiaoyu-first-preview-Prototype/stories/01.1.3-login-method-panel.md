## Common Issues & Fixes

### Issue: Invite Code Does Not Feel Primary

**Symptom**: User notices phone login first.  
**Cause**: Phone group has similar visual weight.  
**Fix**: Give invite code stronger grouping, clearer heading, and lighter phone/Google treatment.

### Issue: Panel Too Tall

**Symptom**: Boundary note is pushed too far down on iPhone SE.  
**Cause**: Excessive padding or overly tall inputs.  
**Fix**: Keep compact but touch-safe 44px controls and reduce secondary group spacing.

### Issue: Button Looks Like Final Submit

**Symptom**: `获取验证码` competes with future primary login action.  
**Cause**: Secondary button too strong.  
**Fix**: Use outlined secondary styling.

## Design Notes

Visual requirements:

- Use restrained surfaces on the dark page.
- Keep input radius at 8px.
- Avoid purple-dominant forms; use purple only for focus/accent.
- Keep text sizes appropriate for dense mobile UI.

UX considerations:

- The user may arrive through targeted promotion, invitation, or an already enabled phone/Google account, so login should feel normal without over-explaining rollout mechanics.
- Phone and Google login exist for account entry/recovery, but DD-001 should not route attention away from the invite-code flow.
- Do not imply photo upload, profile creation, or history creation happens on login.

## Improvement Log

### 2026-05-14 - Normal Login Page Positioning

**Original**: Login panel emphasized invitation and seed-test mechanics (`用邀请账号进入`, `种子测试账号`, `本次种子测试含 1 次个人形象生成额度。`, `没有种子账号时使用。`).

**Improved to**: Normal product login language (`登录 Style AI`, `邀请码`, `首次体验包含 1 次个人形象生成额度。`, `使用手机号接收验证码登录。`, `使用 Google 继续`, `使用 google.com 完成授权登录。`).

**Reason**: Sue clarified that first-batch rollout should not make the page look special or provisional. Restriction can be handled by targeted promotion, invite code, phone whitelist, Google account whitelist, or backend access control.

**Impact**: The page reads as a normal product surface while preserving controlled access.

**Learned**: Keep rollout mechanics out of primary UI copy unless the user needs that information to act.

### 2026-05-14 - Separate Invite Code From Account Login

**Original**: The field was labeled `邀请码或账号`, which implied the product would issue standalone accounts.

**Improved to**: The primary field is only `邀请码`; account login is represented by `手机号登录` and `使用 Google 继续`.

**Reason**: Sue clarified that no separate account credential will be provided. First-batch access can be controlled by invitation scope, invite code, phone whitelist, Google account whitelist, or backend rules.

**Impact**: Avoids a misleading account model while preserving normal login choices.

**Learned**: Keep invite/access-control credentials separate from account login providers.

## Issue Log

### 2026-05-14 - Prefilled Invite Code Input Lost Contrast

**Problem**: Browser autofill/pre-filled input styling rendered the invite code input with a light background, making the text low-contrast against the dark login panel.

**Root Cause**: The Tailwind utility background did not override browser autofill styling for the prefilled text input.

**Solution**: Added scoped input and `:-webkit-autofill` styles under `#start-login-method-panel` so all login inputs keep the dark surface and warm off-white text.

**Impact**: The invite code remains readable in the default prefilled state and the fix is limited to the login method panel.

**Learned**: Prefilled prototype inputs need explicit autofill styling when the page uses a dark visual system.
