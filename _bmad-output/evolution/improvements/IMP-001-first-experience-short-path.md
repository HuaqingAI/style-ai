# Improvement: 首次生成短链路减负

**ID:** IMP-001  
**Type:** UX Improvement  
**Priority:** High  
**Status:** Identified  
**Date:** 2026-05-18  

---

## Party Mode Review Decision

**Review date:** 2026-05-18  
**Review agents:** John (PM), Sally (UX), Winston (Architect), Amelia (Dev)  
**Decision:** Adopted

The review agreed that the modification direction is correct and should become a DD-001 prototype constraint. The core framing is:

> The first experience should not ask the user to configure generation. The user should do less judgment work; the product should provide more judgment.

Adopted constraints:

- 01 is the primary modification scope.
- 02 requires regression review so "change direction" does not reintroduce complex setup.
- 03 requires profile-timing compatibility only; do not force profile creation before the first useful try-on path.
- Remove `01.4-photo-quality-check` and `01.5-generation-setup` from the first happy path, but do not delete their capabilities.
- Compress visible flow; keep system states for quality check, default generation intent, task creation, result viewing, feedback, saving, and variation.
- Result first screen must center on result image, one judgment sentence, and two primary actions: `换个方向` / `保存`.

Open decisions to resolve during Scope/Design:

1. What is the first automatic generation direction, and how is it explained?
2. What are the severe photo-blocking criteria?
3. Is the result judgment sentence generated by AI, rules, or a template-plus-AI field?
4. Does `换个方向` directly generate, or present a small set of lightweight direction choices first?
5. What exactly does `保存` save: image, result, style direction, preference, or profile?
6. After negative feedback, does the product only record the signal, offer regeneration, or adjust the next direction?
7. How does 02 stay low-configuration instead of becoming a hidden version of 01.5?
8. What is 03's minimum usable state when no saved profile exists?

## Opportunity

**What are we improving?**

DD-001 首次真实预览已经完成本地原型验收，但当前 happy path 对首次用户仍有 7 个显性步骤：登录/首页、选照片、质量检查、生成设置、等待、结果。用户的核心动机是快速看到一个像自己的结果，而不是学习质量检查、生成参数、额度、反馈、历史和档案管理。

**Why does this matter?**

首次生成完成率和“像自己 / 可用于判断”反馈是 MVP 的核心验证指标。如果首次链路过长，用户可能在看到价值前就被配置、审核、解释和管理概念拖住。后续二次生成、额度付费和常用形象都依赖第一次结果建立信任。

---

## Data

**Existing Evidence:**

- Product brief 明确：Style AI 的核心不是“给建议”，而是“先让用户看到自己”。
- Trigger map 明确：第一版体验必须围绕“看见我自己”“知道为什么适合/不适合”“照片和档案使用可信”“额度透明”组织。
- Scenario 01 当前 shortest path 有 7 个显性页面。
- DD-001 本地原型已验收通过，但验收对象是完整原型链路，不等于生产首链路的最小摩擦版本。
- Expo 当前实现仍是照片分析工具形态，尚未落地 DD-001，因此还有机会在实现前压缩体验结构。

**User Feedback / Product Direction:**

Sue 提出首次体验应压到最短：`进入 -> 选照片 -> 等待 -> 看结果`。质量检查、生成设置、保存档案、反馈、历史应存在但不作为显性步骤；结果页第一屏先给结果图、一句判断和两个动作。

**Hypothesis:**

如果把首次生成主链路压缩为 4 个用户感知步骤，并把质量检查、生成设置、反馈、档案和额度降级为自动处理、轻提示、折叠区或行为触发，那么首次生成完成率和首次结果可判断率会提升，同时不会牺牲关键的信任边界。

---

## Proposed Solution Direction

**What will we change?**

后续 Scope/Design 阶段重新定义 DD-001：

- 照片上传后直接创建默认生成任务；不再进入独立质量检查页和生成设置页。
- 系统默认生成一个适合日常判断的稳妥方向。
- 照片质量检查变成上传页/任务创建前的自动轻提示；严重不可用才阻断。
- 结果页首屏只保留结果图、一句判断、两个主动作：换个方向、保存。
- 详细原因、避雷项、分析图、反馈、常用形象提示和历史入口全部渐进展示。

## Success Metrics

- 用户感知 happy path 不超过 4 步。
- 选中照片后不要求首次用户再选择风格或填写偏好。
- 轻微照片问题不新增阻断页面。
- 结果页首屏信息不超过：主图、一句判断、换个方向、保存。
- 仍能采集“像不像 / 能不能判断”的核心反馈信号。
- 保持原 MVP 指标：20-50 个种子用户中，至少 60% 完成首次生成并表示“像自己、可用于判断”。

## Prototype Acceptance Criteria

- `AC-DD001-01`: 新用户从登录到首张结果，主路径不得出现独立质量检查页。
- `AC-DD001-02`: 新用户从登录到首张结果，主路径不得出现生成参数设置页。
- `AC-DD001-03`: 上传合格照片后自动进入等待页。
- `AC-DD001-04`: 轻微照片问题不阻断生成，只显示 inline 提示。
- `AC-DD001-05`: 严重照片问题阻断，并给出单一明确动作：重新上传。
- `AC-DD001-06`: 结果页首屏必须包含结果图、一句判断、`换个方向`、`保存`。
- `AC-DD001-07`: 反馈默认不展开；负向或不确定反馈后才展开原因。
- `AC-DD001-08`: 保存、第二次生成、try-on 前才触发资料保存，不在首轮生成前强制出现。
