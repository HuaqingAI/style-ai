# 03.7.1 Credit Purchase Full Page

## Purpose

Build `03.7-credit-purchase.html` as a mobile prototype draft for deciding whether to buy more generation credits after Linman has seen value from one try-on result.

## Objects

- `obj-credit-header`: top navigation with back path to `03.6-history-collection.html`.
- `obj-credit-value-recap`: concise recap of what the previous consumed credit helped decide.
- `obj-credit-balance-card`: current balance, used credit, next generation cost, and urgency copy.
- `obj-credit-plan-list`: selectable plan cards for small, recommended, and stock-up options.
- `obj-credit-policy-note`: purchase and generation consumption boundary copy.
- `obj-credit-purchase-actions`: buy selected plan, decline, continue try-on, and history routes.
- `obj-credit-state-feedback`: visible feedback panel for purchasing, success, error, and declined states.
- `obj-credit-state-controls`: local reviewer controls for six page states.

## State Behavior

- `default`: recommended plan is selected by default; no feedback panel is visible.
- `selected`: user has actively selected a plan; selected summary and primary label update.
- `purchasing`: primary action is disabled and feedback says purchase is being confirmed.
- `success`: newly purchased credits are added to the visible balance; follow-up routes are emphasized.
- `purchase-error`: no payment or credits were changed; retry and decline paths remain available.
- `declined`: user chose not to buy now; page shows return-to-result/history actions.

## Layout Requirements

- Mobile-only single column, max width `428px`.
- Use the same Tailwind CDN setup and dark Style AI surface as `03.2` through `03.6`.
- Keep plan cards compact and stable at 375px; text must not overflow buttons or cards.
- Cards use `rounded-lg`, light borders, compact type scale, and no nested decorative cards.
- Avoid marketing-style hero copy; the page should feel like a decision-support payment step.
- Avoid horizontal overflow at all target widths.

## JavaScript Requirements

- Implement `setCreditState(state)`.
- Implement `selectPlan(planId)`, `purchaseSelectedPlan()`, `declinePurchase()`, and `showCreditToast(message)`.
- Update `data-state`, `data-selected-plan`, `data-purchased`, and `data-balance-after` on `#app`.
- Use `credits`, `tryOnResult`, and `creditPurchase` from `data/demo-data.json` with a complete fallback object in the page.
- Expose `window.__STYLE_AI_PROTOTYPE__` with state setter and action helpers for automated verification.
- Update `03.5` and `03.6` credit buttons so they navigate to `03.7-credit-purchase.html`.

## Acceptance Criteria

Agent-verifiable:
- All object IDs exist.
- Six states can be triggered.
- Three plan buttons exist and selection updates `data-selected-plan`, selected styling, summary copy, and primary action label.
- Purchase success updates `data-purchased=true`, visible balance, and success feedback.
- Purchase error keeps `data-purchased=false` and shows retry copy.
- Declined state shows non-blocking return paths.
- `03.5` and `03.6` credit actions reach `03.7-credit-purchase.html`.
- No console errors at load and during state switching.
- No horizontal overflow at 375px, 393px, and 428px.

User-evaluable:
- The page feels like a natural continuation after seeing value, not a forced paywall.
- The recommended plan and alternatives feel understandable.
- Credit consumption and failed-generation handling feel explicit.
- It is clear that purchasing credits does not automatically start another generation.
