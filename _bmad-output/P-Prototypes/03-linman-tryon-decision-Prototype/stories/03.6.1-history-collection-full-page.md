# 03.6.1 History Collection Full Page

## Purpose

Build `03.6-history-collection.html` as a mobile prototype draft for saving Linman's try-on result, managing comparison/favorite state, and deciding what to do after the result has value.

## Objects

- `obj-history-header`: top navigation with back path to `03.5-tryon-result.html`.
- `obj-history-hero`: concise copy that this result is saved to history and can be compared without saving a common profile.
- `obj-current-saved-result`: current saved cream jacket result with preview, recommendation, credit, saved time, and decision question.
- `obj-history-collection-actions`: compare/favorite actions and visible status chips.
- `obj-compare-list`: comparison list containing current result and recent results.
- `obj-history-decision-actions`: return-to-result, try-another-reference, and credit-plan actions.
- `obj-history-state-controls`: local reviewer controls for five page states.

## State Behavior

- `default`: current result is saved to history, not yet favorited, compare list visible.
- `compare`: current result is marked as in comparison; compare status and row styling update.
- `favorite`: current result is saved and favorited; favorite status and copy update.
- `empty`: no saved try-on result is available; show recovery actions.
- `save-error`: saving failed; explain no extra credit is charged and offer retry.

## Layout Requirements

- Mobile-only single column, max width `428px`.
- Use the same Tailwind CDN setup and dark Style AI surface as `03.2` through `03.5`.
- Keep the current result preview compact so the compare list is reachable on 375px screens.
- Cards use `rounded-lg`, light borders, and compact text scale.
- Do not use nested decorative cards or marketing hero layout.
- Avoid horizontal overflow at all target widths.

## JavaScript Requirements

- Implement `setHistoryState(state)`.
- Implement `toggleCompare()`, `toggleFavorite()`, `retrySave()`, and `showHistoryToast(message)`.
- Update `data-state`, `data-compared`, and `data-favorited` on `#app`.
- Use `tryOnResult`, `recentResults`, and optional `historyCollection` from `data/demo-data.json` with a complete fallback object in the page.
- Keep the `03.7` credit action as a toast placeholder until `03.7-credit-purchase.html` exists.
- Expose `window.__STYLE_AI_PROTOTYPE__` with state setter and action helpers for automated verification.

## Acceptance Criteria

Agent-verifiable:
- All object IDs exist.
- Five states can be triggered.
- Compare and favorite actions update app dataset and visible status chips.
- Empty state hides saved-result content and shows recovery actions.
- Save-error state shows retry and does not show duplicate credit charge copy.
- `03.5` result actions can reach `03.6-history-collection.html`.
- No console errors at load and during state switching.
- No horizontal overflow at 375px, 393px, and 428px.

User-evaluable:
- The page feels like a decision-support history workspace, not a generic gallery.
- The boundary between history result, favorite, comparison, and common profile is clear.
- The next actions feel useful after one consumed credit.
- The screen keeps purchase/credit entry available without feeling like a forced upsell.
